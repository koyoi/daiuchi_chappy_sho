# Classic 探索アルゴリズム (kishi-to-classic)

Classic エンジンの探索部の設計と改良履歴。
ソースコード: `src/engine.cpp` / `src/engine.h`

---

## 探索の全体像

```text
chooseMove()
  ├── 定石照合 (序盤8手以内)
  ├── 詰み探索 (最大7手, 持ち時間の10%)
  └── 反復深化 (depth 1 → depthLimit)
        └── scoreRootMovesParallel()
              └── search() — αβ探索
                    └── quiescence() — 静止探索
```

1. **定石照合**: `book.txt` からハッシュ引きで候補手を選択（手数 <= 8）
2. **詰み探索**: `MateSolver` が王手のみの手を生成し、7手詰めまでを探索
3. **反復深化**: depth 1 から順に深くし、制限時間 (`shouldStop()`) を超えたら直前の完了した深さの最善手を採用
4. **ルート並列**: 各ルート手を複数スレッドで並列に探索

---

## αβ探索 (`search()`)

Minimax + Alpha-Beta 枝刈り。`board.side == rootSide` で最大化/最小化を分岐。

### パラメータ

| パラメータ | 説明 |
|-----------|------|
| `depth` | 残り探索深さ |
| `ply` | ルートからの距離 (0始まり) |
| `alpha`, `beta` | 探索窓 |
| `rootSide` | エンジン側の手番 |
| `allowNullMove` | Null Move Pruning を許可するか |

### 探索フロー

```text
1. 置換表プローブ → ヒットすれば即返し、TT最善手を記憶
2. 合法手生成 → 空なら詰み/ステイルメイト判定
3. depth <= 0 なら静止探索へ
4. 王手判定
5. Null Move Pruning (条件付き)
6. IID: TT最善手がなく depth >= 5 なら浅い予備探索で最善手候補を取得
7. 手の並び替え (TT最善手 > 駒取り > 王手 > 成り > Killer > Counter > History)
8. Futility Pruning (浅い深さで静かな手をスキップ)
9. 各手を探索:
   a. Check Extension (王手をかける手で depth+1)
   b. PVS (最初の手は全窓、2手目以降は null window)
   c. Late Move Reduction (遅い静かな手を縮小探索)
   d. 再探索 (LMR/PVS null window が有望なら全深さ・全窓で)
   e. αβ更新
   f. β-cutoff → Killer/History/Counter Move 更新
10. 置換表に結果を格納
```

---

## 静止探索 (`quiescence()`)

depth 0 に達した後、戦術的に不安定な局面（駒交換の途中など）で評価値が大きくぶれるのを防ぐ。

- **対象手**: 駒取り、成り、王手（depth >= 4 の場合のみ）
- **最大深さ**: 6手 (`QuiescenceDepth`)
- **Standing Pat**: 静的評価値をベースラインとし、それを超える手のみ探索
- **Delta Pruning**: マージン 1400 で明らかに改善しない手をカット

---

## 置換表 (Transposition Table)

手順違いで同じ局面に合流した場合に、以前の探索結果を再利用する。

- **サイズ**: 2^20 = 1,048,576 エントリ
- **ハッシュ**: Zobrist ハッシュ (盤面 + 持ち駒 + 手番を XOR で差分更新)
- **排他制御**: 64 本のストライプ mutex
- **世代管理**: `ttGeneration_` で古いエントリを自然に上書き
- **格納情報**: ハッシュ鍵、深さ、スコア、フラグ (Exact/Lower/Upper)、最善手

---

## 手の並び替え (Move Ordering)

枝刈り効率は手の探索順序に大きく依存する。良い手を先に探索すれば、αβ枝刈りでより多くの不要な手をスキップできる。

### 優先順位

| 優先度 | カテゴリ | スコア |
|--------|---------|--------|
| 1 | 置換表の最善手 | 20000 |
| 2 | 駒取り (MVV-LVA) | 10000 + 取った駒の価値×10 - 動いた駒の価値 |
| 3 | 王手を与える手 | 8000 |
| 4 | 成り | 6000 |
| 5 | Killer Move (第1スロット) | 5000 |
| 6 | Killer Move (第2スロット) | 4900 |
| 7 | 駒打ち | 駒価値/3 + 敵玉近接ボーナス |
| 8 | 静かな盤上移動 | History Heuristic + 犠牲手ボーナス |

### MVV-LVA (Most Valuable Victim - Least Valuable Attacker)

高価値の駒を低価値の駒で取る手を優先。例: 歩で飛車を取る手 > 飛車で歩を取る手。

### 犠牲手ボーナス

相手の利きに入る非駒取り手で、敵玉のチェビシェフ距離 2 以内に迫る場合にボーナス加点 (+300)。「ただ捨ての後の王手飛車取り」のような手を早期に探索させる。

## 枝刈りテクニック

### Null Move Pruning (NMP)

「パス（何もしない手）」を打って相手に手番を渡し、それでもなお自分が有利なら、通常の手を指せばさらに有利になるはずなので、この局面を枝刈りする。

- **条件**: `ply > 0`, 王手されていない, `depth >= 3`
- **縮小**: R = 3 (パス後の探索は depth - 4 で行う)
- **二重パス防止**: 連続パスを禁止 (`allowNullMove = false`)
- **将棋での有効性**: 駒打ちがあるため完全なツークツワンクが起きにくく、NMP の誤判定リスクが低い

### Late Move Reduction (LMR)

手の並び替え後、後ろの方にある静かな手（駒取り・成り・王手でない手）は有望でない可能性が高いため、まず縮小した深さで探索し、有望なら全深さで再探索する。

- **条件**: `depth >= 3`, 5手目以降 (`moveIndex >= 4`), 静かな手, 王手でない
- **縮小量**: R = 1 + (depth >= 6 ? 1 : 0) + (moveIndex >= 10 ? 1 : 0)
- **再探索**: 最大化側で `val > alpha`、最小化側で `val < beta` の場合

ルートでも別途 Root LMR を適用: 上位 `RootPruneWidth` 手（デフォルト 15）以降を depth - 3 で探索。

### Futility Pruning

浅い深さ (depth 1-2) で、静的評価値にマージンを加えてもα（最大化側）に届かない場合、静かな非王手の手をスキップ。

- **マージン**: depth 1 → 400, depth 2 → 900
- **条件**: 王手されていない、静かな手、王手を与えない
- **安全弁**: 全手スキップ時は静的評価値を返す

### αβ枝刈り

αβ探索の基本: `alpha >= beta` で部分木を丸ごとカット。手の並び替えが良いほどカット率が上がり、同じ時間でより深く読める。

## ヒューリスティック

### Killer Move Heuristic

β-cutoff を引き起こした静かな手（駒取り・成りでない手）を記憶し、同じ深さの兄弟ノードで優先的に試す。

- **テーブル**: `killers_[MaxPly][2]` — 各 ply に 2 スロット
- **保存条件**: β-cutoff かつ駒取り/成りでない手
- **FIFO**: 新しい Killer が古い方を押し出す

### History Heuristic

過去の探索でβ-cutoff を引き起こした手を統計的に記録し、move ordering に反映する。

- **テーブル**: `history_[2][81][81]` — 手番 × 移動元 × 移動先 (int16_t)
- **更新**: cutoff した手に `+depth^2`、cutoff 前に試した静かな手に `-depth^2` (malus)
- **Gravity**: `entry += bonus - entry * |bonus| / 16384` — 自然に古い情報を減衰
- **aging**: 反復深化のイテレーション間で全値を半減

## ルート並列探索

複数スレッドでルート手を並列に探索する。

### 共有αβ窓

`std::atomic<int> sharedAlpha` で全スレッドが最善スコアを共有。各スレッドは探索開始時に共有αを読み取り、より良いスコアが見つかれば CAS (Compare-And-Swap) で更新。

これにより、あるスレッドが良い手を見つけると、他のスレッドが即座にそのαを利用してより効率的に枝刈りできる。以前は各スレッドが `[-MateScore, +MateScore]` のフルウィンドウで探索しており、枝刈りが全く効いていなかった。

### ワークスティーリング

`std::atomic_int nextIndex` でルート手のインデックスを管理し、各スレッドがアトミックにインクリメントして次の手を取得。先に終わったスレッドは次の手を自動的に処理する。

## 改良履歴

コードから読み取れる現在の実装状況と、直近の探索改良。

### これまでに実装済みの基盤

- **Bitboard**: 81ビット (uint64_t + uint32_t) による高速な駒位置・利き管理
- **事前計算テーブル**: 短距離駒の攻撃テーブル、飛び駒のスライド関数
- **Zobrist ハッシュ**: 盤面・持ち駒・手番を XOR で差分更新
- **駒割りの差分更新**: `applyMove()` で material score をインクリメンタルに維持
- **反復深化**: depth 1 から順に深化、時間制限で中断
- **置換表**: 2^20 エントリ、64 ストライプ mutex、世代管理
- **静止探索**: 駒取り・成り・王手を追加探索 (最大6手)、デルタ枝刈り
- **詰み探索**: MateSolver による7手詰め探索（持ち時間の10%を使用）
- **Root LMR**: ルートで上位 N 手以外を縮小探索
- **評価特徴量**: 100次元（駒数、利き、玉の安全性、ピン、打ち込み脅威、両取り、駒の連携など）
- **MLP 評価**: 100→128→64→1 の多層パーセプトロンによるリーフ評価（要再学習）
- **オンライン学習**: 対局結果に基づく重みの自動更新
- **定石データベース**: `book.txt` によるハッシュ引き（序盤8手）
- **序盤安全性**: 序盤で罠に嵌まりやすい手にペナルティ
- **マルチスレッド**: ルート手を複数 CPU で並列探索

### 2025-06 探索の全面改良

同じ持ち時間で到達深さが 2〜4 手向上（5秒で depth 6 程度 → depth 8 程度）。

| 改良 | 効果 |
|------|------|
| Null Move Pruning | パスしても有利な局面を即枝刈り。探索木を大幅に縮小 |
| Killer Move Heuristic | 同じ深さで良かった静かな手を兄弟ノードで優先。cutoff 率向上 |
| History Heuristic (+ malus) | 過去の探索統計による静かな手の順序改善。長期的な学習効果 |
| 置換表最善手の move ordering 利用 | 以前は TT に最善手を格納していたが ordering に未使用だった。最優先で試すように |
| Late Move Reduction (内部ノード) | 以前はルートのみ。全ノードで後ろの静かな手を縮小探索 |
| Futility Pruning | 浅い depth で見込みのない静かな手をスキップ |
| ルート並列探索の共有α窓 | 以前は各スレッドがフルウィンドウで探索。共有αで枝刈り効率を劇的改善 |
| 犠牲手の move ordering ボーナス | 敵玉付近への捨て駒を早期に探索し、王手飛車等の戦術を発見しやすく |

### 2025-06 探索・評価の第2次改良

探索効率のさらなる改善と、評価関数の戦術的精度向上。

| 改良 | 効果 |
|------|------|
| Aspiration Windows | 反復深化で前回スコア ±50 の狭窓から探索。窓外なら全窓で再探索。探索ノード削減 |
| Principal Variation Search (PVS) | 最善候補のみ全窓、2手目以降は null window で試行。失敗時のみ再探索。10〜20% ノード削減 |
| Check Extension | 王手をかける手で探索を1手延長。水平線効果を軽減し、戦術的読みを深化 |
| Internal Iterative Deepening (IID) | TT に最善手がない深い局面 (depth >= 5) で浅い予備探索を実行。手の並びを改善 |
| Counter Move Heuristic | 相手の直前の手に対する反撃手を記憶 (β-cutoff 時)。move ordering ボーナス +4800 |
| 両取り検出特徴量 (feature 98) | 1つの駒が2つ以上の高価値敵駒を攻撃する関係を検出。戦術的形勢判断に寄与 |
| 駒の連携特徴量 (feature 99) | 自駒同士が互いを守り合う度合い（coordination）。囲いの堅さを数値化 |
