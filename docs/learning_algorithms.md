## 2. Classic エンジン — 線形モデルと MLP

Classic エンジンは2つの評価方式を持つ：シンプルな線形モデル（重み付き和）と、
より表現力のある MLP（多層パーセプトロン）。

### 2.1 特徴量設計（100次元）

局面から100個の数値特徴を抽出する（`src/evaluation.cpp`）。
先手視点で計算し、後手番なら符号を反転する。

| 特徴量番号 | 内容 | 備考 |
|-----------|------|------|
| 0–20 | 駒数、PST（駒位置テーブル）、持ち駒、玉近接度 | 基本的な駒の配置評価 |
| 21–27 | 玉間距離、手番、局面フェーズ（序盤/中盤/終盤） | ゲーム進行度 |
| 28–31 | 脅威値（攻撃/浮き駒/守られた駒）、/100で正規化 | 駒の安全性 |
| 32–35 | 玉の安全性（攻め駒数、守り駒数、逃げ道、王手状態） | 玉の危険度 |
| 36–41 | 戦術的要約（王手/取り/成り/合法手数、玉周り攻撃）| 重い特徴量あり |
| 51–54 | 玉周りの金・銀・味方・敵の駒数 | 囲いの厚み |
| 55–57 | 歩の構造（二歩、孤立歩、パス歩候補） | 歩の形 |
| 58–60 | 飛車の活用度（素通し筋、半素通し筋、玉筋配置） | 飛車の利き |
| 61–63 | 陣地制圧（自陣、中央、敵陣の支配度） | 盤面の勢力圏 |
| 64–67 | 駒の機動力（飛車/10、角/10、銀、香） | 利きの広さ |
| 90–97 | フェーズゲート特徴量（終盤係数 × 主要安全性特徴量） | 終盤で重要度が変わる |

**Bitboard による高速計算**: 81ビットのビットボード（`uint64_t lo` + `uint32_t hi`）で
利き計算や攻め駒カウントを効率的に行う。

### 2.2 線形評価

最もシンプルな評価関数。100個の特徴量と100個の重みの内積を取る：

```
score = Σ(feature[i] × weight[i])   (i = 0..99)
```

### 2.3 MLP 評価

3層のニューラルネットワークで、線形モデルより複雑な特徴量の組み合わせを学習できる。

**アーキテクチャ** (`tools/mlp_eval.py`):

```
入力(100) → Linear(256) → BatchNorm → LeakyReLU(0.01) → Dropout(0.1)
         → Linear(128) → BatchNorm → LeakyReLU(0.01) → Dropout(0.05)
         → Linear(1)   → (生のロジット出力)
```

**C++ 推論時の計算** (`src/evaluation.cpp`):

```
h1 = LeakyReLU(W1 × features + b1)   // 100→256
h2 = LeakyReLU(W2 × h1 + b2)         // 256→128
logit = W3 × h2 + b3                  // 128→1
score = logit × 361                    // センチポーン(cp)に変換
```

学習時は BatchNorm を使うが、推論時はエクスポート時に **BatchNorm 融合** で消去する（後述）。

### 2.4 オンライン学習（対局中の学習）

`OnlineLearner` は対局中にリアルタイムで重みを更新する（`src/learning.cpp`）。

**データ記録** (`recordMove`):

各局面で以下を記録する：
- 選ばれた手の特徴量ベクトル
- 全合法手の特徴量の平均ベクトル

**更新式** (`finishGame` — 対局終了時):

```
diff = 選んだ手の特徴量 − 合法手の平均特徴量

sign = (勝ちなら+1, 負けなら-1)
     × (エンジンの手なら 1.0, 相手の手なら 0.7)
     × recency（古い局面 0.5 → 新しい局面 1.0 で線形スケール）

weight[i] += lr × sign × diff[i]     (lr = 0.002)
```

更新後、全重みに weight decay (0.9995) を適用し、[-5.0, 5.0] にクランプする。

**直感的な解釈**: 「勝った試合で選んだ手は、平均的な手より良い特徴を持っていたはず」
という仮定のもと、選んだ手と平均手の特徴量の差分方向に重みを動かす。
方策勾配法（Policy Gradient）に似た考え方。

### 2.5 MLP のオンライン学習

MLP にも対局中の学習がある（`finishGameMlp`）。

**ラベル生成（Eval Bootstrap）**:

```
smoothResult = 勝ちなら 0.8, 負けなら 0.2
label = 0.6 × σ(eval / 361) + 0.4 × smoothResult
label = clamp(label, 0.01, 0.99)
```

- `eval`: その局面でのエンジン評価値
- `σ`: シグモイド関数
- 勝敗の生ラベル (0/1) ではなく、エンジン評価値を混ぜた「ソフトラベル」を使う

**バックプロパゲーション**: BCEWithLogitsLoss 相当の勾配を各層に逆伝播する。
学習率 0.0005、weight decay 0.9999、重みクランプ [-3.0, 3.0]。

### 2.6 MLP のバッチ学習パイプライン

大量の棋譜データからオフラインで MLP を学習する（`tools/train_mlp.py`）。

**学習パイプライン**:

```
棋譜(CSA形式) → csa_parser.py で解析
    ↓
局面を抽出（30手以上の対局、1局面おきにサンプリング）
    ↓
[オプション] mcts_label.py で強いエンジンの評価値をラベルに使用
    ↓
Classic エンジンで100特徴量を抽出 → TSVファイル
    ↓
mlp_eval.py で PyTorch 学習
    ↓
export_mlp.py で BatchNorm 融合 → テキスト重みファイル
```

**PyTorch 学習設定**:

| 項目 | 値 |
|------|-----|
| 損失関数 | BCEWithLogitsLoss（シグモイド + 二値交差エントロピー） |
| オプティマイザ | Adam (lr=0.001, weight_decay=1e-4) |
| スケジューラ | CosineAnnealingLR |
| バッチサイズ | 4096 |
| エポック数 | 30 |
| データ分割 | 訓練90% / 検証10% |

### 2.7 BatchNorm 融合（推論の高速化）

学習時には BatchNorm を使って勾配の流れを安定化するが、
推論時には前の Linear 層に統合して計算を1回で済ませる（`tools/export_mlp.py`）。

**数学的導出**:

```
Linear層:    y = Wx + b
BatchNorm層: z = γ × (y − μ) / √(σ² + ε) + β

融合後: z = W_fused × x + b_fused

ここで:
  scale = γ / √(σ² + ε)
  W_fused = scale × W        （各出力ニューロンの重みにscaleを乗算）
  b_fused = scale × (b − μ) + β
```

融合後は BatchNorm の running_mean / running_var を保持する必要がなく、
単純な行列演算だけで推論できる。

## 3. NNUE エンジン — 差分更新可能なニューラルネットワーク

NNUE (Efficiently Updatable Neural Network) は Stockfish などのチェスエンジンで
実績のある手法を将棋に適用したもの。
巨大な入力次元を持つが、1手ごとに変化する特徴量だけを差分更新することで
高速な推論を実現する。

### 3.1 HalfKP 特徴量（170,662次元）

**Half King-Piece（HalfKP）** とは、「自玉の位置」と「盤上の駒の位置・種類」の
組み合わせで特徴量を定義する方式。

```
ボード特徴量: 170,586 = 81(玉位置) × 26(色付き駒種) × 81(駒位置)
持ち駒特徴量:      76 = 2(先手/後手) × 38(持ち駒インデックス)
合計:         170,662次元
```

**特徴量インデックスの計算** (`src/nnue.cpp`):

```
index = kingSquare × (26 × 81) + coloredPieceType × 81 + pieceSquare
```

ある局面で「先手玉が59にいて、後手の角が22にいる」場合、
これが1つの特徴量インデックスとして表現される。

170,662次元のうち、1局面でアクティブな特徴量は約30–40個だけ。
つまり入力はほぼすべてゼロで **極めてスパース** である。

### 3.2 ネットワーク構造

```
入力(170,662) → L0: Linear(512) → SCReLU → 出力(1)
```

ただし、先手視点と後手視点の両方を計算して結合する：

```
先手アキュムレータ(512) ─┐
                        ├→ SCReLU → 結合(1024) → L1: Linear(1) → 評価値
後手アキュムレータ(512) ─┘
```

**SCReLU（Squared Clipped ReLU）活性化関数**:

```
SCReLU(x) = clamp(x, 0, 1)²
```

通常の ReLU と比べて、二乗することでより滑らかな活性化を得る。
整数演算でも効率的に計算できる。

### 3.3 差分更新（アキュムレータ）

NNUE の核心技術。1手進めるとき、全入力を再計算するのではなく、
変化した特徴量だけを足し引きする。

**フル計算**（初期化時や玉が動いたとき）:

```
accumulator[i] = bias[i] + Σ weight[i][active_feature]
                            (全アクティブ特徴量について)
```

**差分更新**（通常の1手）:

```
// 駒が移動した場合、古い位置の特徴を引き、新しい位置の特徴を足す
accumulator[i] -= weight[i][removed_feature]
accumulator[i] += weight[i][added_feature]
```

1手で変わる特徴量は通常2–4個程度なので、
170,662次元の全計算に比べて **数千倍高速** になる。

玉が動くと関連する全ての HalfKP 特徴量が変わるため、
その視点のアキュムレータは全再計算が必要。

### 3.4 量子化（整数推論）

浮動小数点演算を避け、int16/int32 の整数演算だけで推論する。

**量子化スケール**:

```
QA = 255   // アキュムレータの量子化スケール
QB = 64    // 出力の量子化スケール
```

**重みの量子化**:

```
L0 重み: float → int16 (×QA して丸め)
L0 バイアス: float → int16
L1 重み: float → int32 (×QA×QA×QB して丸め)  ← SCReLU の二乗を考慮
L1 バイアス: float → int32
```

**推論結果のデスケール**:

```
output = Σ(L1_weight[i] × SCReLU_quantized(accumulator[i]))
score = output / (QA × QA × QB)    // = output / 4,161,600
```

### 3.5 SIMD 最適化

大量の評価を高速に行うために、CPU の SIMD 命令を活用する（`src/nnue.cpp`）。

**AVX2**（16個の int16 を同時処理）:

```cpp
// 512ユニットを32回のループで処理（512 / 16 = 32）
for (int i = 0; i < L0Size; i += 16) {
    __m256i acc = _mm256_load_si256(accumulator + i);
    acc = _mm256_max_epi16(acc, zero);      // clamp 下限
    acc = _mm256_min_epi16(acc, qa_vec);    // clamp 上限
    __m256i sq = _mm256_madd_epi16(acc, acc); // 二乗 + 隣接加算
    sum = _mm256_add_epi32(sum, sq);
}
```

SSE4.1（8個同時）、スカラー（1個ずつ）のフォールバックもある。

### 3.6 バイナリフォーマット（NNU4）

```
[0-3]   マジック "NNU4" (4バイト)
[4-7]   L0Size = 512 (int32)
[8+]    L0重み: 512 × 170,662 × 2バイト (int16) ≒ 175MB
        L0バイアス: 512 × 2バイト
        L1重み: 1024 × 4バイト (int32)
        L1バイアス: 4バイト
```

### 3.7 PyTorch での学習

**スパーステンソルによる効率的学習** (`tools/train_nnue.py`):

170,662次元の密な入力テンソルを作るとメモリが爆発する。
代わりに `torch.sparse_coo_tensor` でスパーステンソルを作り、
`torch.sparse.mm` で L0 層の行列演算を行う。

```python
# 密な行列演算: input(batch×170662) × weight.T(170662×512) → メモリ不足
# スパース: アクティブな特徴量のインデックスと値1.0だけ保持
sparse_input = torch.sparse_coo_tensor(indices, values, (batch, 170662))
hidden = torch.sparse.mm(sparse_input, l0_weight.T)
```

**損失関数（シグモイド交差エントロピー）**:

```python
SCALE = 361.0  # 評価値→確率の変換温度

pred = σ(output / SCALE)
loss = −(target × log(pred) + (1−target) × log(1−pred))
```

**Eval Bootstrap**:

```python
LAMBDA = 0.75

label = LAMBDA × σ(engine_eval / 361) + (1−LAMBDA) × game_result
```

エンジンの局面評価値と実際の勝敗を 75:25 でブレンドする。
純粋な勝敗ラベルはノイズが多い（勝勢の局面でも後の悪手で負けることがある）ため、
評価値を混ぜることで局面の品質をより正確に反映する。

**学習設定**:

| 項目 | 値 |
|------|-----|
| オプティマイザ | Adam (lr=0.001, weight_decay=1e-6) |
| スケジューラ | CosineAnnealingLR |
| バッチサイズ | 8192 |
| 混合精度 | AMP (FP16) で GPU 高速化 |

## 4. Alpha エンジン — AlphaZero 方式（ResNet-SE + MCTS）

AlphaZero の手法を将棋に適用したエンジン。
深層ニューラルネットワークが「方策（次の手の確率分布）」と「価値（勝率）」を
同時に出力し、MCTS（モンテカルロ木探索）で探索を行う。

### 4.1 入力特徴量（45チャンネル × 9×9）

局面を 45 枚の 9×9 画像として表現する（`src/alpha_onnx_inference.cpp`）。

| チャンネル | 内容 | 値 |
|-----------|------|-----|
| 0–13 | 先手の駒14種 | 0/1（その駒がある場所が1） |
| 14–27 | 後手の駒14種 | 同上 |
| 28–34 | 先手の持ち駒7種 | count / maxCount（正規化） |
| 35–41 | 後手の持ち駒7種 | 同上 |
| 42 | 先手の攻め数 | attackCount / 8（各マス） |
| 43 | 後手の攻め数 | 同上 |
| 44 | 手番 | 先手番なら全1.0、後手番なら全0.0 |

持ち駒は盤面と異なり「全マスに同じ値を埋める」形式で、
枚数を最大値で割って正規化する（歩の最大=18、飛車の最大=2 など）。

### 4.2 ResNet-SE ネットワーク構造

**全体構造** (`tools/train_alpha.py`):

```
入力(45ch) → Conv2d(3×3, 192ch) → BatchNorm → ReLU
    ↓
[SEブロック × 15]  ← 残差接続 + Squeeze-and-Excitation
    ↓
┌─ 方策ヘッド ─────────────────────────┐
│ Conv2d(1×1, 32ch) → BN → ReLU       │
│ → Flatten → Linear(32×81, 2187)      │
│ → 2187個のロジット                    │
└──────────────────────────────────────┘
    ↓
┌─ 価値ヘッド ─────────────────────────┐
│ Conv2d(1×1, 1ch) → BN → ReLU        │
│ → Flatten → Linear(81, 256) → ReLU  │
│ → Linear(256, 3)                     │
│ → WDL (勝ち/引き分け/負け) 確率       │
└──────────────────────────────────────┘
```

**SE（Squeeze-and-Excitation）ブロック**:

チャンネルごとの重要度を学習する注意機構。

```
入力 x
  ↓
Conv(3×3) → BN → ReLU → Conv(3×3) → BN → 中間出力
  ↓
[Squeeze] 全マスの平均を取る (9×9 → 1×1)  → (192次元ベクトル)
  ↓
[Excitation] FC(192→48) → ReLU → FC(48→192) → Sigmoid → (チャンネル重み)
  ↓
中間出力 × チャンネル重み  (チャンネルごとにスケーリング)
  ↓
+ x（残差接続） → ReLU → 出力
```

SE は「どのチャンネル（特徴マップ）が今の局面で重要か」を動的に判断する。
たとえば「飛車の利き」チャンネルは飛車がある局面で重要度が高くなる。

**方策出力のエンコーディング（2187 = 81 × 27）**:

各マスに対して27種類の指し手を定義する：

```
チャンネル 0–9:   移動元の方向（10方向、成りなし）
チャンネル 10–19: 移動元の方向（10方向、成りあり）
チャンネル 20–26: 駒打ち（7種類の駒）
```

### 4.3 改良 MCTS

標準的な MCTS にいくつかの改良を加えている（`src/alpha_mcts.cpp`）。

**PUCT 選択（動的 c_puct）**:

```
c_puct = log((1 + N + 19652) / 19652) + 2.5

UCB_score = Q(s,a) + c_puct × P(s,a) × √N(s) / (1 + N(s,a))
```

- `Q(s,a)`: 行動価値（平均勝率）
- `P(s,a)`: NN が出力した事前方策
- `N(s)`: 親の訪問回数
- `N(s,a)`: この手の訪問回数
- c_puct は訪問回数の増加に伴い対数的に増加し、深い探索で探索幅を広げる

**FPU（First Play Urgency）**:

未訪問のノードの Q 値を `親のQ − 0.2` に設定する。
親がすでに良い手を見つけているとき、未探索の手を過度に優遇しない。

**温度スケジュール**:

```
手数 ≤ 30: 温度 1.0 で訪問回数に比例してランダムサンプリング
手数 > 30: 温度 0（訪問回数最大の手を決定的に選択）
```

序盤に多様性を持たせ、中終盤では最善手を選ぶ。

**早期打ち切り**:

シミュレーション全体の 25% 以上を消費した時点で、
最善手の訪問比率が 90% を超えたら探索を終了する。

**木の再利用（Tree Reuse）**:

相手の着手後、MCTS 木の中から Zobrist ハッシュで該当するサブツリーを探し、
それをルートに昇格させる。前回の探索で蓄積した統計情報を再利用できる。

### 4.4 教師あり学習（Supervised Learning）

Floodgate（ネット将棋サーバー）の R3000 以上の強豪棋譜から学習する
（`tools/train_alpha.py`）。

**損失関数**:

```
policy_loss = CrossEntropy(予測方策, 実際に指された手)
value_loss  = CrossEntropy(予測WDL, 対局結果)
total_loss  = policy_loss + value_loss
```

**学習設定**:

| 項目 | 値 |
|------|-----|
| オプティマイザ | SGD (lr=0.01, momentum=0.9, weight_decay=1e-4) |
| スケジューラ | CosineAnnealingLR |

### 4.5 強化学習（Self-Play RL）

教師あり学習で得たモデルを出発点として、自己対戦で更に強化する。

**自己対戦データの生成** (`tools/alpha_self_play.py`):

```
1. 現在のモデルで自己対戦（200局 × 400シミュレーション/手）
2. 各局面の MCTS 訪問分布を記録（getvisits コマンドで取得）
3. 対局結果を WDL ラベルとして全局面に付与
```

**RL 学習** (`tools/train_alpha_rl.py`):

```python
# 方策損失: MCTS訪問分布と NN 出力の KL ダイバージェンス
policy_loss = KL_div(NN出力の方策, MCTS訪問分布)

# 価値損失: WDL の交差エントロピー
value_loss = CrossEntropy(NN出力のWDL, 対局結果WDL)

total_loss = policy_loss + value_loss
```

ここが教師あり学習との大きな違い：
- 教師あり学習: 「人間/強豪が指した1手」を正解とする
- 強化学習: 「MCTS が探索した訪問分布」を正解とする（より豊富な情報）

**リプレイバッファ**:

最大200万局面を保持する FIFO バッファ。
古い局面は新しい局面に置き換えられる。学習時はランダムサンプリング。

**RL ループ** (`tools/alpha_train_loop.py`):

```
for iteration in 1..N:
    1. 自己対戦: 200局、400 sims/手
    2. リプレイバッファに追加
    3. バッファからサンプリングして学習（1エポック、lr=0.001）
    4. 評価: 新モデル vs 現最強モデル（20局、200 sims）
    5. 勝率 ≥ 55% なら採用、そうでなければ棄却
```

ゲート（採用判定）を設けることで、偶然の劣化を防ぐ。

### 4.6 知識蒸留（Knowledge Distillation）

大きなモデル（教師）の知識を小さなモデル（生徒）に転写する
（`tools/train_alpha_small.py`）。

```
教師: 15ブロック / 192チャンネル（大きい・遅い・強い）
生徒: 10ブロック / 128チャンネル（小さい・速い）
```

**蒸留損失**:

```python
# ソフトターゲット（教師の出力、温度T=3.0で軟化）
teacher_policy = softmax(teacher_logits / T)
policy_soft = KL_div(log_softmax(student_logits / T), teacher_policy) × T²

# ハードターゲット（実際の正解ラベル）
policy_hard = CrossEntropy(student_logits, 正解の手)

# ブレンド（α=0.7: 教師70%, 正解30%）
policy_loss = 0.7 × policy_soft + 0.3 × policy_hard
```

温度 T=3.0 で教師の出力確率を「軟化」させ、
2番手・3番手の手に対する評価情報も生徒に伝える。
（温度1.0だと最善手の確率が支配的で、他の手の情報がほぼ消える）

### 4.7 INT8 量子化

CPU 推論を高速化するための量子化（`tools/quantize_alpha.py`）。

```
1. 代表的な局面データ（約1000局面）でキャリブレーション
2. 各層の活性化値の範囲を統計的に測定
3. FP32 重み → INT8 重みに変換（スケールとゼロポイントを設定）
4. ONNX 形式で出力
```

FP32 → INT8 で約2–3倍の CPU 推論高速化が期待できる。

## 5. 横断的なテクニック

### 5.1 Eval Bootstrap（評価値ブートストラップ）

全エンジンに共通する重要テクニック。学習ラベルとして
「対局の勝敗」と「エンジンの局面評価値」を混ぜる。

```
label = λ × σ(eval / 361) + (1−λ) × game_result
```

**なぜ必要か**: 対局の勝敗は究極的な正解だが、ノイズが多い。
勝勢の局面でも後の悪手1つで負けることがある。
エンジン評価値を混ぜることで、局面そのものの良し悪しをより正確に学習できる。

| エンジン | λ の値 | 用途 |
|---------|--------|------|
| Classic（オンライン） | 0.6 | MLP のオンライン更新 |
| NNUE | 0.75 | PyTorch 学習 |
| Classic（クロストレーニング） | 0.7 | 他エンジンの評価値を利用 |

### 5.2 MCTS ラベリング

Classic の MLP 学習で、より強いエンジン（MCTS/Alpha/NNUE）の評価値を
訓練ラベルとして使う技術（`tools/mcts_label.py`）。

```
1. 各局面を強いエンジンに評価させる（movetime 指定）
2. 評価値をシグモイドで勝率に変換: label = σ(score / 361)
3. ゲーム結果とブレンド: final = blend × mcts_label + (1−blend) × result
```

弱いエンジンが強いエンジンの「知識」を吸収する、
一種の **知識蒸留（データレベル）** と言える。

### 5.3 クロスエンジン学習

異なるエンジン間で知識を共有する手法（`tools/cross_train_loop.py`）。

```
Classic で自己対戦 → 局面を収集
    ↓
NNUE/Alpha エンジンで各局面をラベリング（強い評価値を取得）
    ↓
Classic MLP を学習
    ↓
改善されたか自己対戦で評価（勝率52%以上で採用）
    ↓
繰り返し
```

### 5.4 損失関数の比較

| エンジン | 損失関数 | 理由 |
|---------|---------|------|
| Classic MLP | BCEWithLogitsLoss | 勝率（0–1）を予測する二値分類 |
| NNUE | Sigmoid CrossEntropy (scale=361) | 評価値→勝率変換のスケールを合わせる |
| Alpha 教師あり | CrossEntropy（方策 + WDL） | 手の分類 + 勝敗の分類 |
| Alpha RL | KL Divergence（方策） + CE（WDL） | MCTS分布との差異を最小化 |
| 蒸留 | KL Div（soft） + CE（hard）のブレンド | 教師のソフト出力と正解の両方から学習 |

## 付録: 用語集

| 用語 | 説明 |
|------|------|
| αβ探索 | ミニマックス探索の高速化手法。不要な枝を刈る |
| MCTS | モンテカルロ木探索。シミュレーションで最善手を推定 |
| PUCT | MCTS の選択式。探索と活用のバランスを取る |
| HalfKP | 玉の位置と駒の位置・種類の組み合わせで特徴量を定義 |
| SCReLU | Squared Clipped ReLU。`clamp(x,0,1)²` |
| BatchNorm | ミニバッチの統計量で正規化。学習を安定化 |
| Eval Bootstrap | エンジン評価値と勝敗を混ぜたラベル |
| SE (Squeeze-and-Excitation) | チャンネル注意機構。重要な特徴マップを強調 |
| WDL | Win/Draw/Loss。3値で勝率を表現 |
| FPU | First Play Urgency。未訪問ノードの初期値 |
| 知識蒸留 | 大きなモデルの出力を小さなモデルに教える |
| スパーステンソル | ほぼゼロの巨大テンソルを効率的に表現 |
| 量子化 | 浮動小数点→整数に変換して推論を高速化 |
| SFEN | 将棋の局面表記法（Shogi Forsyth-Edwards Notation） |
